# GitHub App Integration API

Backend-managed GitHub integration endpoints for setup callbacks, webhook processing, project-scoped access requests, and explicit repository linking.

## Integration Model (Current)

- GitHub App installation authorization is stored independently from repository linkage.
- Project-to-installation authorization is stored in `project_github_installation_authorizations`.
- Project-to-repository linkage is stored in `project_repositories` and is explicit (`POST /api/supervisor/projects/{projectId}/github/link`).
- Frontend never calls GitHub directly.

## Public Endpoints (`/api/github/*`)

These endpoints are intentionally `permitAll` in `SecurityConfig` because they are used by GitHub callback/webhooks or public request tokens.

### GET /api/github/setup

Processes GitHub App setup callback.

Query params:

- `installation_id` (required)
- `state` (optional, but required to complete request-token flow)

Behavior:

- stores/updates installation in `github_app_installations`
- records project-level installation authorization in `project_github_installation_authorizations`
- does **not** auto-link repository to project
- supports:
  - request-token callback flow (`state` generated by backend)
  - legacy base64/base64url state fallback (`{ "projectId": "..." }`)
- issues `303 See Other` redirect to frontend

Redirect outcomes:

- request-flow success:
  - `{FRONTEND_BASE_URL}/github/access-updated?token={resultToken}&status=success`
- direct setup success:
  - `{FRONTEND_BASE_URL}/supervisor/projects/{projectId}?tab=overview&githubSetup=success&installationId={installationId}`
  - includes `githubAccessUpdated=true` when request-flow context exists
- failure:
  - project-aware fallback when possible: `/supervisor/projects/{projectId}?tab=overview&githubSetup=failed`
  - otherwise: `/github/access-updated?status=failed`

### POST /api/github/webhooks

Processes GitHub webhooks with HMAC verification.

Headers:

- `X-GitHub-Event` (required)
- `X-Hub-Signature-256` (required)

Behavior:

- verifies signature using `GITHUB_APP_WEBHOOK_SECRET`
- supports `installation` and `installation_repositories` events
- updates installation lifecycle/status metadata
- on installation delete/suspend style states, clears project installation authorization linkage

### GET /api/github/access-requests/validate?token=...

Validates a public project-scoped request token.

Response data:

- `projectId`
- `projectTitle`
- `status` (`PENDING`)
- `expiresAt`

### POST /api/github/access-requests/continue?token=...

Continues public request-token flow and returns GitHub authorize/install URL.

Response data:

- `projectId`
- `githubAuthorizeUrl`

### GET /api/github/access-updated/summary?token=...

Loads post-callback summary for the public success page.

Response data:

- `projectId`
- `projectTitle`
- `installationId`
- `accessScope` (`NO_REPOSITORIES`, `SINGLE_REPOSITORY`, `MULTIPLE_REPOSITORIES`)
- `accessibleRepositoryCount`
- `repositories[]` (`repositoryId`, `name`, `fullName`, `url`, `ownerLogin`, `defaultBranch`)

### POST /api/github/access-updated/acknowledge?token=...

Acknowledge/close public callback result token.

Behavior:

- marks acknowledgement timestamp
- clears one-time result token fields

## Supervisor Endpoints Related to GitHub App Flow

All endpoints below are authenticated and require role `SUPERVISOR`.

### GET /api/supervisor/projects/{projectId}/github/installations/{installationId}/repositories?page=&size=

Returns paginated repositories accessible under a specific installation for a specific project authorization context.

Rules:

- validates project ownership
- validates installation usability (active/not deleted/suspended)
- validates project-level installation authorization exists and matches supervisor identity
- `page` must be `>= 1`
- `size` defaults/caps from config:
  - `app.github.installation-repositories.default-page-size`
  - `app.github.installation-repositories.max-page-size`

Response data:

- `items[]` (`repositoryId`, `name`, `fullName`, `url`, `ownerLogin`, `defaultBranch`)
- `page`, `size`, `returnedCount`, `totalCount`
- `hasNext`, `hasPrevious`, `nextPage`

### POST /api/supervisor/projects/{projectId}/github/link

Explicitly links one selected installation repository to one project.

Request body:

```json
{
  "installationId": 118120112,
  "repositoryId": 123456789
}
```

Behavior:

- validates project ownership + installation authorization
- verifies repository exists under installation (iterates installation pages until found)
- updates primary project repository linkage fields (`installation_id`, `repository_external_id`, URL/name/owner/default branch)
- records linkage audit (`linked_by_supervisor_user_id`, `linked_at`)
- attempts immediate refresh sync

### POST /api/supervisor/projects/{projectId}/github/access/remove

Removes project-level GitHub access authorization and linked repository/cache data for that project context.

### POST /api/supervisor/projects/{projectId}/github/access-requests

Creates short-lived project-scoped access request token/link.

### GET /api/supervisor/projects/{projectId}/github/access-requests/validate?token=...

Supervisor-authenticated token validation variant (project + requester-bound).

### POST /api/supervisor/projects/{projectId}/github/access-requests/continue?token=...

Supervisor-authenticated continuation variant; returns GitHub authorize/install URL.

### POST /api/supervisor/projects/{projectId}/github/refresh

Refreshes linked repository cache (metadata/commits/contributors).

Installation-disconnect handling:

- if installation token creation fails with disconnected statuses (`401/403/404/422` style), refresh raises domain error
- project linkage and cached GitHub data are cleared for the project to avoid stale broken linkage

## Shared Dashboard Endpoints (Supervisor + Student)

- `GET /api/{role}/projects/{projectId}/github`
- `GET /api/{role}/projects/{projectId}/github/activity?page=&size=`
- `GET /api/{role}/projects/{projectId}/github/contributors?page=&size=`

Where `{role}` is `supervisor` or `student`.

Data source:

- `project_repositories`
- `project_repository_commits`
- `project_repository_contributors`

Main project detail endpoints also include `github` preview block and access metadata:

- `authorizedInstallationId`
- `accessibleRepositoryCount`
- `accessScope`

## Scheduled Jobs

Configured under `app.github.jobs`:

- `access-request-cleanup` (fixed-delay scheduler):
  - deletes expired pending/expired request records
  - clears expired result tokens
- `repository-refresh` (cron scheduler):
  - periodically refreshes linked primary GitHub repositories
  - batch size, cron, and zone are env-configurable
